# HU Push/Fold Chart (≤15 BB short-stack)

This document is the **authoritative source-of-truth** for `poker/strategy/data/push_fold_hu.json`. Mirrors the format of `hu_preflop_chart_README.md` so the two charts (deep 100 BB and short-stack push/fold) share a common review pattern.

## Background

Stacks below ~15 BB effective change correct play fundamentally: implied odds collapse, postflop SPR is near zero, and the EV of medium-sized raises drops below pure push (jam) or fold. At these depths, the strategy table that ships in `preflop_100bb_hu.json` is mis-calibrated — it emits standard raise sizes that are structurally bad when the bot is committing 30-40% of its stack on a single preflop raise.

The short-stack heuristic in `poker/strategy/short_stack.py` already does the soft version of the fix — it suppresses medium-raise probability mass and redistributes to jam/fold below 20 BB. This chart is the **hard version**: an explicit Nash-style push/fold lookup that bypasses the deep-stack table entirely below 15 BB.

## Stack depth buckets

| Depth | Source rationale |
|---|---|
| 5 BB | Almost any-two-cards regime; SB jams ~80%+ |
| 7 BB | Top of the push-fold sweet spot in HU |
| 10 BB | Standard textbook depth; widely-published reference ranges |
| 12 BB | Transition zone — some hands still call/raise normally |
| 15 BB | Upper bound; above this the deep-stack table takes over |

## Scenarios in scope

Two decision contexts in HU push/fold:

| Scenario | Position | Hero faces | Hero acts |
|---|---|---|---|
| `sb_open` | `SB` | (acts first) | jam or fold |
| `bb_vs_jam` | `BB` | SB jam | call or fold |

This matches the canonical "SB jams or folds; BB calls or folds" decomposition that all Nash push/fold solvers compute. No 3-bet handling — at these depths a 3-bet IS an all-in.

## Action vocabulary

Only two actions per row:

| Action | Meaning |
|---|---|
| `jam` | All-in for whatever the effective stack is |
| `fold` | Fold |
| `call` | (BB only, vs SB jam) Call the all-in |

Per-row probabilities sum to 1.0. Binary 100/0 frequencies for v1; mixed strategies can come later when calibration against true Nash output reveals which hands need mixing.

## v1 calibration status

**The current ranges are approximate Nash equilibrium estimates, not solver output.** They're seeded from textbook references (Sklansky-Chubukov push tables, public Nash push/fold approximations published on training sites) but have NOT been verified against a clean Nash solve.

Known approximations:
- Ranges are coarsened at the borderlands (no mixing — every hand is 100% one action)
- Stack-depth granularity is buckets, not continuous (a real solver would interpolate)
- Single-table SNG (chip EV) assumption — no ICM adjustments (cash WTA-SNG context per the project's tournament mode)

Calibration roadmap (deferred, not blocking ship):
1. Pull canonical Nash push/fold ranges from a trusted source (HoldemResources, ICMIZER public output, the Wizard of Odds tables, or a clean PioSolver HU run)
2. Diff against the v1 placeholder ranges
3. Update json + this README's border-flip log

For each border-flip applied during initial authoring, document the hand + source disagreement in the section below.

## Range targets (chart-level invariants)

Approximate aggregate frequencies the chart aims to hit. Tests in `tests/test_strategy/test_push_fold_hu.py` assert against these bands.

| Depth | SB jam % | BB call % |
|---|---|---|
| 5 BB | 70-90% | 40-55% |
| 7 BB | 50-65% | 25-35% |
| 10 BB | 40-50% | 18-25% |
| 12 BB | 30-40% | 14-20% |
| 15 BB | 22-32% | 10-16% |

Hand-class shape (v1 default per depth bucket): pairs first → suited aces → offsuit aces → suited Broadway → suited connectors → high-card offsuit. As stack shrinks, ranges widen from the top.

## Sources consulted (v1 authoring)

- Approximate Sklansky-Chubukov push tables (general "any two cards" thresholds)
- General Nash push/fold theory as commonly described in poker training literature
- The HU preflop chart README (`hu_preflop_chart_README.md`) for canonical-hand conventions

The v1 ranges were generated by `poker/strategy/data/generate_push_fold_hu.py` using deterministic rules. Re-run that script after edits.

## What's NOT in this chart

- **Multi-way push/fold.** v1 is HU only. Multi-way short-stack scenarios use the legacy `short_stack.py` heuristic until calibrated data is added.
- **ICM-adjusted ranges.** WTA SNG = chip EV throughout, so ICM never applies for the current game mode.
- **Mixed frequencies.** Same v1 limitation as the deep-stack chart — binary 100/0 per hand. Calibration pass.
- **Stack-depth interpolation.** Lookup picks the nearest bucket; no smooth interpolation between depths. Sufficient for v1 since the lookup is gated to <15 BB.
