# Cash Mode — Ephemeral Tourists

> **Why this exists.** Today's casino-spawn flow (`cash_mode/casino_provisioning.py`) requires `archetype='fish'` personalities to already have an `ai_bankroll_state` row in the sandbox before `load_fish_ids` will return them. In a fresh sandbox, no fish have ever been seated, the set is empty, and **no casino ever spawns** — the classic cold-start chicken-and-egg the sim runner's `initial_bank_pool_seed` hack works around. The web app has no such escape hatch.
>
> This spec replaces the persistent-fish model with **ephemeral tourists**: synthetic, on-demand fish personalities generated by a factory when the casino spawns, vanishing when the table tears down. No bankroll seeding, no tourist injection, no chicken-and-egg. Each tourist also carries one designated **leak** — a specific, identifiable exploit — so casinos become a "read the table" puzzle rather than a generic loss machine.

## Goals

1. **Casinos auto-spawn in cold sandboxes** the moment the bank pool clears the threshold ($2: 5,000 chips). No prerequisite seating.
2. **Each casino feels like rotating real tourists** — fresh names + caricatures every spawn, not a recycled cast.
3. **Each tourist has one diagnosable leak** drawn from a per-template pool, so the same template can produce different exploits at different casinos.
4. **The conservation invariant holds** — chips in seats at teardown return to the pool via a new `casino_seat_return` ledger reason.
5. **No relationship-system pollution** — ephemeral tourists don't accumulate relationship history with grinders.

## Design

### Source of truth for fish

| Before | After |
|---|---|
| 8 hand-curated `archetype='fish'` personalities in `personalities.json` | 8 templated archetypes in `cash_mode/tourist_factory.py`, instantiated on demand |
| Fish must have `ai_bankroll_state` row to be eligible | No bankroll required — fish are stateless, chips live only on the seat |
| `load_fish_ids(bankroll_repo, sandbox_id)` returns persistent ids | `generate_tourist_batch(rng, count)` returns fresh `TouristProfile`s |
| `resolve_tourist_injection` tops up depleted fish bankrolls | Deleted — fish never have bankrolls |

The 8 JSON personalities are **kept** (they still work outside casinos for any future use; they serve as the reference implementation that the templates mirror) but no longer gate-keep casino spawn.

### Tourist anatomy

```python
@dataclass(frozen=True)
class TouristProfile:
    personality_id: str            # "tourist-<uuid8>" — synthetic, stable for table lifetime
    display_name: str              # "Marge from Jersey"
    template_key: str              # "vacation_dad" — for capture/dossier aggregation
    personality_dict: Dict         # full personality config, stashed inline in the seat
```

The `personality_dict` mirrors the existing fish JSON structure (anchors, verbal_tics, physical_tics, default_confidence/attitude, rule_strategy='fish', staker/borrower=unwilling) plus two new fields: `ephemeral: True` and `fish_leak: "<leak_key>"`.

### Designated leak system

The base fish rule strategy (`poker/rule_strategies.py:527`) already plays bad poker — calling station, raises nothing, folds to large bets only with marginal hands. The **leak** layers one specific deviation on top, making each tourist exploitable in an identifiable, repeatable way.

#### Leak catalogue

```python
class FishLeak(str, Enum):
    CALLS_DOWN_TOP_PAIR        = "calls_down_top_pair"        # never folds top pair, value-bet thin
    CHASES_ANY_DRAW            = "chases_any_draw"            # pays bad price for FDs / OESDs
    DOESNT_BELIEVE_BIG_BETS    = "doesnt_believe_big_bets"    # hero-calls large bets light
    LIMPS_EVERY_HAND           = "limps_every_hand"           # preflop never folds
    POT_COMMITTED_EARLY        = "pot_committed_early"        # once 30%+ in, can't fold
    OVERVALUES_FACE_CARDS      = "overvalues_face_cards"      # KJo, Q9o, anything with paint
    CALLS_RIVER_LIGHT          = "calls_river_light"          # river sizing concept missing
    SPITE_RAISES_WHEN_LOSING   = "spite_raises_when_losing"   # tilt min-raises as bluffs
```

Each leak modifies exactly one branch of `_strategy_fish`. The base behavior is preserved when no leak applies — the leak fires only when its trigger condition is met, leaving every other decision identical to the calling-station baseline.

#### Per-template leak pools

Each template carries a `candidate_leaks: List[FishLeak]` (3–4 entries, all in-character). At spawn time, the factory picks one via `rng.choice`. **Two tourists from the same template at different casinos can have different leaks**, so grinders can't memorize `template → leak`; they have to actually read the hands.

#### Fish fingerprint invariant

Every template's anchors must satisfy the fish profile, enforced by a unit test:

```python
def test_all_templates_satisfy_fish_fingerprint():
    for t in TOURIST_TEMPLATES:
        a = t.anchors
        assert 0.70 <= a["baseline_looseness"] <= 0.95
        assert 0.10 <= a["baseline_aggression"] <= 0.45
        assert a["adaptation_bias"] == 0.0    # never learns — defining trait
        assert a["recovery_rate"] == 0.0      # stays tilted
        assert a["poise"] <= 0.45             # rattle-able
        assert len(t.candidate_leaks) >= 2    # variance required
        assert len(set(t.candidate_leaks)) == len(t.candidate_leaks)
```

This makes it impossible to accidentally add a "fish" template that plays well or has only one possible leak.

### Seat shape

The seat dict (`cash_tables.seats_json[i]`) carries the tourist's full inline personality so controller construction doesn't need a DB lookup:

```python
def ai_slot_ephemeral(profile: TouristProfile, chips: int) -> Dict[str, Any]:
    return {
        "kind": "ai",
        "personality_id": profile.personality_id,
        "chips": int(chips),
        "display_name": profile.display_name,
        "ephemeral_personality": profile.personality_dict,
    }
```

A new `personality_for_seat(seat, personality_repo)` helper checks the inline dict first, falls back to repo lookup for non-ephemeral seats. Every existing call site that does `personality_repo.load(seat["personality_id"])` is routed through this helper.

### Conservation: residual stack at teardown

When a casino is deleted, any seat with `chips > 0` writes a `casino_seat_return` ledger row — the mirror of `casino_seat_seed`. Reason added to `BANK_POOL_DEPOSIT_REASONS` so pool depth correctly absorbs returned chips.

```python
def record_casino_seat_return(
    repo, *, personality_id: str, amount: int, sandbox_id: str, context: dict,
) -> None:
    """Mirror of record_casino_seat_seed — returns leftover seat chips to pool."""
```

The conservation invariant in `chip_ledger_audit.py` (`drift == 0`) continues to hold: every chip moved is paired with a ledger entry.

### Relationship suppression

At every `memory_manager.set_relationship_repo(...)` call site, check the table's `table_type`. If `'casino'`, skip the wire-up. `_process_relationship_events` in `poker/memory/memory_manager.py:147` already silent-no-ops when `_relationship_repo is None`, so no detector changes are needed.

Critically, `set_relationship_repo` also assigns `opponent_model_manager._relationship_repo` internally (`memory_manager.py:263`), which means skipping it ALSO suppresses:
- Chat-intent relationship events (gated by `opponent_manager.has_relationship_repo` in `flask_app/handlers/chat_relationship.py:63`)
- `apply_cash_pair_pnl` writes (gated by the same `_relationship_repo is None` check; only ever called from `dispatch_events`)
- Stack-dominance axis writes (same gate)

Affected call sites:
- `flask_app/routes/cash_routes.py:800` (was :793 — line shifted)
- `flask_app/routes/game_routes.py:595, :1327` (were :594, :1326)
- `flask_app/routes/experiment_routes.py:2822` (only matters if experiments ever spin casino tables — leave alone for now)

**Caveat: staking-event writes are NOT suppressed by this approach.** `cash_mode/player_staking.py` and `cash_mode/ai_carry_resolution.py` construct their own `OpponentModelManager(relationship_repo=...)` directly and call `record_event` for `STAKE_OFFERED`/`STAKE_REPAID`/`STAKE_DEFAULTED`/`STAKE_FORGIVEN`. For casino tourists this is **self-suppressing by data invariant**: every tourist template has `staker_profile.willing=False` and `borrower_profile.willing=False`, so the staking resolution code never reaches `record_event` for tourist pids. If we ever change that invariant, suppression would need to extend to the staking paths.

## Implementation order

Each step is independently shippable; later steps depend on earlier ones.

1. **Ledger reason** (`core/economy/ledger.py`). New `casino_seat_return` deposit reason + helper. Smallest, isolated, lets you write the teardown-return test first.
2. **`FishLeak` enum + extended `_strategy_fish`** (`poker/rule_strategies.py`). One unit test per leak. No callers consume the leak yet, but the strategy is ready.
3. **Context plumbing**. `_build_rule_context` gains the new fields (`fish_leak`, `street`, etc.). Existing tests should pass unchanged; new fields just default-None for non-fish.
4. **Tourist factory** (`cash_mode/tourist_factory.py`). Templates + factory + fingerprint tests. Pure module, no integration yet.
5. **Seat helpers** (`cash_mode/tables.py`). `ai_slot_ephemeral` + `personality_for_seat`. Update call sites that look up by pid.
6. **Casino provisioning swap** (`cash_mode/casino_provisioning.py`). Replaces fish-loading with factory; ledger row carries template/leak in context; teardown writes return rows.
7. **Relationship suppression** (cash + game route call sites).
8. **Delete tourist injection** (`cash_mode/closed_economy.py`) + dead tests.
9. **Cold-start integration test** — fresh sandbox to spawned casino, end-to-end.

Steps 1–4 can be parallelized across agents if desired. Step 5 onwards is sequential.

## Open questions

1. **Where does `FishLeak` live?** Options: (a) `poker/rule_strategies.py` next to the fish strategy that consumes it, (b) `cash_mode/tourist_factory.py` next to the templates that assign it, (c) standalone `poker/fish_leaks.py`. Recommend (a) — the strategy is the authoritative consumer; factory imports.
2. **Allow `FishLeak.NONE` in some pools?** Would mean ~10–20% of tourists are "boring fish" — base strategy, no specific exploit. Makes diagnosis harder (grinder doesn't know whether a leak exists). Recommend: **defer**. The base fish strategy is already exploitable; add NONE if grinders find casinos trivial.
3. **Allow multiple leaks per tourist?** Compound leaks could be added later via `leaks: List[FishLeak]`. YAGNI for v1 — one leak keeps the puzzle clean.
4. **Per-template name pools vs single shared pool?** This spec uses per-template pools (Brenda fits bachelorette, not retired_know_it_all). Costs slightly more curation. Alternative: shared pool with type-agnostic names. Recommend per-template — character coherence matters more than name variety.
5. **Should grinders' dossier surface leak diagnoses?** "You've seen Marge call river light 4 times — likely `CALLS_RIVER_LIGHT`." This is a natural follow-on that requires session-memory aggregation; out of scope for v1, easy to layer in after.
6. **Casino spawn threshold** — currently $2 needs 5,000 chips in the pool. Untouched by this spec. If cold-start sandboxes find the pool fills too slowly (vice deposits are the only source for now), revisit threshold or add a one-time bootstrap deposit at sandbox creation.

## Validation: how we know this worked

After implementation:

- **Cold-start test**: create a fresh sandbox, run vice resolution until pool ≥ 5,000, refresh lobby once, verify a $2 casino spawns with 2–4 ephemeral tourists, each carrying a `fish_leak` from its template's pool.
- **Conservation test**: spawn casino, manually delete via provisioning (force teardown), verify `chip_ledger_audit` reports `drift == 0`. Residual chips at the seats must show up in the bank pool reserve calculation.
- **Variety test**: spawn 20 casinos with 4 tourists each (80 tourists total). Assert at least 6 of the 8 templates appear, and at least 5 distinct `FishLeak` values appear in the aggregate.
- **No relationship pollution**: play 10 hands at a casino table, query `cash_pair_stats` and `relationship_events`, expect zero new rows tied to any `tourist-*` pid.
- **Backward compat**: the 8 hand-curated fish personalities still work in non-casino contexts (lobby tables, experiments) — their `fish_leak` is `None`, base fish strategy runs.

## Out of scope (followups)

- **Compound leaks** (multiple leaks per tourist)
- **Leak-difficulty tiers** (some leaks exploit-harder than others; casinos could weight pools by stake)
- **Dossier auto-diagnosis** of tourist leaks for grinders
- **$10 casino tier** with stronger fish (less leaky, more chips). Templates could split into `casino_tier='$2'` vs `'$10'` pools.
- **Real fish in lobby tables** — the existing 8 personalities staying is largely decorative; if we want them to actually show up at lobby tables, separate work.
- **Hungry-grinder demand gate** from the dev-branch v2 (`551e0e51`) — independent improvement, doesn't conflict with this spec.
