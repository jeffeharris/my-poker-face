# The limper detour — building the wrong instrument, then the right one

A long session. It started as routine calibration cleanup and ended somewhere more
interesting: we built a feature, proved it had no edge, retired it, and only *then*
realised we'd been building the wrong thing the whole time. The honest version of
how that happened is worth keeping.

## Where it started (calibration cleanup)

Picked up the archetype-calibration handoff and cleared the backlog, all merged:

- **#314** cheap hygiene — fixed a flaky probabilistic test, hoisted the duplicated
  chart-gen helpers into `_chart_gen.py`, archived a stale handoff. (The hoist was
  verifiable to a zero chart diff, which is the kind of refactor I like.)
- **#315** the `vs_open` intent-tag refactor — retired the implicit "infer
  value/bluff from the raise weight" cliff. The interesting bit was an audit finding
  the obvious shape (an `intent` key *inside* the cell) would crash ~12 consumers
  that do `sum(cell.values())`; the safe shape was a sibling `vs_open_intent`
  section the runtime loader never looks at. Depth charts regenerated byte-identical
  — the migration was provably behaviour-preserving.
- **#316** per-opener `vs_squeeze` key, then **#319** width-scaled squeeze defense
  (so the *squeezer's* range width finally moves the response — wider BB squeeze →
  defend wider). Both clean.

Good, contained work. Then the user pointed at the chart-opportunity census's #1
gap and said: build the next chart.

## The wrong turn

The census flagged **"short-stack first-in over a limper"** with the tag
`pushfold_fallthrough` — its instrumentation noticed that `_try_push_fold_6max`
*bails* when a limper sits in front, dropping the spot to the deep chart at 10-15bb.
So I built exactly that: a push/fold routing fix (#321), gated off.

Then I let the sim do the talking, and it was unkind in a useful way:
- **#322** — naive turn-on vs a never-folding limper: **−7.7 bb/100**. No fold
  equity. The same lesson the reshove taught.
- **#323** — added a fold-equity gate (generalised the reshove's
  `_opponent_fold_equity_ok`). Leak closed → neutral vs a sticky field.
- **#325** — built a *foldy* limper (`FishLeak.LIMP_FOLD`) to measure the upside…
  and there wasn't one: even ungated, vs a generous folder, the iso was **−1.7 to
  −2.8 bb/100**. The ~2.5bb of limp dead money doesn't pay for jamming a wide range
  into a field that wakes up. So I retired the idea: keep it off, ship it as a safe
  gated no-op + the measurement that killed it.

I was pleased with that — a clean negative result, "measure before you build the
chart." And it was clean. It was also answering the wrong question.

## The correction

The user, on reading the writeup: *"I thought that's what we were building. I don't
know how we got stuck on 15bb… I literally thought we were building a
detection/exploit pair."*

They were right, and it's worth being precise about how we drifted. The census
metric is tied to **one code path** (the push/fold lookup), so the only slice of
"exploit limpers" it could *see* was the ≤15bb push/fold fall-through. I built the
symptom the instrument reported, literally — a push/fold chart, which is hard-gated
to 15bb — instead of stepping back to the player's-eye view: *a skilled reg attacks
a weak limper*, and that's a deeper-stack, **iso-raise**, **per-opponent** play, not
a 15bb jam. The push/fold version wasn't a smaller version of the real thing; it was
a different, worse thing that happened to share the word "limper."

Two more things the user caught that I'd have missed:
- *"Are we push/folding for ALL AIs at 15bb? that's not realistic."* — Correct
  instinct, and the code already agreed: push/fold is opt-in on **8 analytical-shark
  personas**, not the field. The "all AIs" I'd been seeing was a **sim artifact**
  (the harness builds controllers via `__new__`, bypassing `__init__`, so the gate
  defaults on for measurement). Production realism was fine; I'd been reading a test
  scaffold as production.

## The right instrument

So we scoped and started the real thing — **punish limpers** as an exploitation-
layer detect/exploit pair (`LIMP_EXPLOIT.md`), same machinery as `vs3bet_exploit`:

- **Detect:** a single *foldy habitual* limper — high vpip−pfr gap (enters
  passively), and explicitly **not** a station/jammer (those call the iso → no fold
  equity). Reuses `classify_opponent_archetype`; no read → no-op.
- **Exploit:** iso-raise wider — shift `fold→open` for **playable** hands only
  (suited or pairs; never opens offsuit junk — widening an open is riskier than the
  vs3bet narrowing direction, so this guard is load-bearing). Skill-graded
  `limp_exploit` knob (shark 0.85 … rec 0.0), flag `LIMP_EXPLOIT_ENABLED` off.
- The foldy-limper opponent from the *failed* push/fold detour (#325) is exactly the
  validation tool for this — so the wrong turn left something reusable.

Built, unit-tested (14 tests), flag off. Next is the validation: does it actually
print vs a foldy field at 25-40bb, and stay neutral vs a station — the upside the
push/fold version never had because it was at the wrong depth with the wrong action.

## The throughline

The recurring lesson this session: **an instrument frames the problem in its own
terms.** The census sees push/fold fall-throughs, so it described a limper-
exploitation opportunity as a push/fold gap — and I built a push/fold gap. The fix
wasn't more rigor inside that frame (the sims were rigorous, and correctly said "no
edge"); it was the user stepping outside it to ask what a *player* would actually do.
A negative result can be both honest and beside the point. Worth remembering when the
next census or probe hands me a tidy-looking gap.
